# Security Scan — 2026-03-05

## Scan Summary

- **Total findings**: 14
- **By type**: 9 bugs, 4 improvements, 0 features, 0 debt
- **By severity**: 1 critical, 4 high, 5 medium, 4 low

## Key Findings

### Critical
- `.env.agent-swarm` committed to git with live DeepSeek API key and Telegram bot token

### High
- Webhook verify tokens (WhatsApp/Meta) use `==` instead of `secrets.compare_digest()` — timing attack
- No rate limiting on web login, registration, password reset, or MFA endpoints
- Vendor sessions stored in in-memory dict (lost on restart, not shared across workers)
- Vendor auth cookies hardcoded `secure=True` (breaks dev, inconsistent with admin portal)

### Medium
- MFA tokens not invalidated after use — replay within 5-min window
- TOTP codes replayable within 30-sec window (no used-code tracking)
- Default DB credentials in config.py fallback
- Legal service uses unsafe `getattr()` for ordering instead of `apply_ordering()`
- API login rate limit too generous (100/min global vs needed 5/min auth-specific)
- Temp passwords for new users don't force password change

### Positive Security Patterns Confirmed
- No SQL injection found — proper parameterized queries throughout
- No eval/exec, no pickle, no subprocess with user input
- Password hashing uses passlib (pbkdf2_sha256/bcrypt)
- TOTP `valid_window=0` is strict
- Token generation uses `secrets.token_urlsafe()`
- Storage has proper path traversal prevention
- CSRF now properly validated on survey endpoints (was previously a known gap)
- Auth CRUD router correctly restricted to admin role only
